## 问题概述

新增的菜单功能（白名单、租户管理、存证管理、监控管理、接口管理）出现两个关键错误：
1. **HTTP 404错误** - API接口无法访问
2. **HTTP 500错误** - 前端页面无法加载

## 问题1：HTTP 404错误（已解决）

### 根本原因

**Clash代理配置导致Docker容器间通信失败**

Docker Daemon的代理配置（`~/.docker/config.json`）中的`noProxy`列表不包含内部容器名称，导致Nginx容器尝试通过代理（127.0.0.1:7890）访问Gateway容器时连接失败。

### 错误链路

```
浏览器 → Nginx (manage-web0)
           ↓
      proxy_pass http://gateway0:8080
           ↓
      检查环境变量：HTTP_PROXY=http://127.0.0.1:7890
           ↓
      检查no_proxy：localhost,127.0.0.1
           ↓
      "gateway0" 不在no_proxy列表中
           ↓
      尝试通过代理连接：http://127.0.0.1:7890 → gateway0:8080
           ↓
      失败！（代理在容器内不存在）
           ↓
      返回 HTTP 404
```

### 解决方案

#### 步骤1：修复Docker代理配置

编辑 `~/.docker/config.json`：

```json
{
  "proxies": {
    "default": {
      "httpProxy": "http://127.0.0.1:7890",
      "httpsProxy": "http://127.0.0.1:7890",
      "noProxy": "*"  // ← 禁用容器内的所有代理
    }
  }
}
```

#### 步骤2：重新创建所有容器

```bash
cd ~/github/primihub-deploy/docker-all-in-one

# 停止并删除受影响的容器
docker stop manage-web0 manage-web1 manage-web2 gateway0 application0
docker rm manage-web0 manage-web1 manage-web2 gateway0 application0

# 重新创建所有服务
docker compose up -d
```

#### 步骤3：清除用户登录token

```bash
docker exec redis redis-cli -a primihub KEYS "sys_user:login_token_*" | \
  xargs -I {} docker exec redis redis-cli -a primihub DEL {}
```

## 问题2：HTTP 500错误（已解决）

### 根本原因

**Nginx配置文件中的根目录路径错误**

容器重建后，nginx配置文件中的`root`路径指向了不存在的目录。

### 错误详情

```
2026/01/10 10:10:13 [error] 30#30: *5 rewrite or internal redirection cycle
while internally redirecting to "/index.html", client: 100.64.0.17,
server: localhost, request: "GET / HTTP/1.1"
```

**错误配置**：
```nginx
location / {
    root   /usr/local/nginx/html;  # ← 目录不存在
    index  index.html index.htm;
    try_files $uri $uri/ /index.html;
}
```

**实际文件位置**：
- 前端文件在：`/usr/share/nginx/html/`
- 配置指向：`/usr/local/nginx/html/`（不存在）

### 解决方案

#### 修复所有3个Nginx容器

```bash
# 修复 manage-web0
docker exec manage-web0 bash -c 'cat /etc/nginx/conf.d/default.conf | \
  sed "s|/usr/local/nginx/html|/usr/share/nginx/html|g" > /tmp/default.conf && \
  cat /tmp/default.conf > /etc/nginx/conf.d/default.conf && \
  nginx -s reload'

# 修复 manage-web1
docker exec manage-web1 bash -c 'cat /etc/nginx/conf.d/default.conf | \
  sed "s|/usr/local/nginx/html|/usr/share/nginx/html|g" > /tmp/default.conf && \
  cat /tmp/default.conf > /etc/nginx/conf.d/default.conf && \
  nginx -s reload'

# 修复 manage-web2
docker exec manage-web2 bash -c 'cat /etc/nginx/conf.d/default.conf | \
  sed "s|/usr/local/nginx/html|/usr/share/nginx/html|g" > /tmp/default.conf && \
  cat /tmp/default.conf > /etc/nginx/conf.d/default.conf && \
  nginx -s reload'
```

## 核心教训

### 1. 容器内部不应该有代理配置

- 代理适用于：容器访问外部互联网
- 代理不适用于：容器间内部通信

配置原则：
```json
{
  "httpProxy": "http://127.0.0.1:7890",  // 用于外网访问
  "noProxy": "*"  // 禁用容器内所有代理，让内部通信直连
}
```

### 2. 容器重建 vs 容器重启

- **重启容器**：`docker restart` - 环境变量不变
- **重新创建容器**：`docker compose down && docker compose up` - 环境变量重新读取

修改Docker配置后，必须**重新创建容器**才能生效！

### 3. Nginx配置路径必须正确

容器重建可能导致配置文件被重置，需要验证：
```bash
# 检查配置
docker exec manage-web0 cat /etc/nginx/conf.d/default.conf

# 检查文件是否存在
docker exec manage-web0 ls -la /usr/share/nginx/html/
```
