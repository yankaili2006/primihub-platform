# 新增模块404问题解决方案

## 问题描述

当添加新的功能模块（如白名单、租户管理、存证管理等）后，前端访问新模块的API接口时报错：**请求出错(404)**

即使后端代码、数据库、路由配置都正确，仍然会出现404错误。

## 问题原因分析

这是一个**多层缓存**导致的问题，涉及以下几个方面：

### 1. Gateway路由缓存问题
- **现象**: Gateway容器使用的是旧镜像，不包含新模块的路由配置
- **原因**: 在`gateway/src/main/resources/application.yaml`中添加了新路由配置，但没有重新编译gateway模块并重建Docker镜像

### 2. Application JAR缓存问题
- **现象**: Application容器中的JAR文件不包含新的Controller类
- **原因**: 添加了新的Controller、Service等类后，没有重新编译相关模块并重建Docker镜像

### 3. Redis权限缓存问题
- **现象**: 用户token中的权限列表不包含新模块的权限
- **原因**: Redis中缓存了旧的权限数据（`sys_auth:bfs_list`和用户登录token）

### 4. 用户Session缓存问题
- **现象**: 已登录用户的session中不包含新权限
- **原因**: 用户登录时获取的权限列表被缓存在Redis中，新增权限后旧token仍然有效

## 完整解决方案

### 步骤1: 确认数据库配置

检查新模块的权限是否已正确配置到数据库：

```bash
# 检查权限是否分配给管理员角色
for db in privacy1 privacy2 privacy3; do
  echo "=== Database: $db ==="
  docker exec mysql mysql -uroot -proot $db -e "
SELECT a.auth_code, COUNT(ra.id) as role_assigned
FROM sys_auth a
LEFT JOIN sys_ra ra ON a.auth_id = ra.auth_id AND ra.role_id = 1 AND ra.is_del = 0
WHERE a.auth_code IN ('Whitelist', 'Tenant', 'Evidence', 'Monitor', 'ApiManage', 'SystemConfig', 'Log')
GROUP BY a.auth_code;"
done
```

如果权限未分配，需要执行相应的SQL脚本（如`grant_permissions.sql`）。

### 步骤2: 重新编译并构建后端模块

```bash
# 进入项目目录
cd /home/primihub/primihub-platform/primihub-service

# 重新编译biz和application模块
mvn clean install -DskipTests -pl biz,application -am

# 重新编译gateway模块（如果gateway配置有变化）
mvn clean install -DskipTests -pl gateway -am
```

### 步骤3: 重新构建Docker镜像

```bash
cd /home/primihub/primihub-platform/primihub-service

# 重新构建镜像
docker build -t 192.168.99.10/primihub/privacy:1.8.0 -f Dockerfile.ubuntu .
```

### 步骤4: 重启相关容器

```bash
cd ~/github/primihub-deploy/docker-all-in-one

# 重启application容器
docker compose stop application0
docker compose rm -f application0
docker compose up -d application0

# 重启gateway容器
docker compose stop gateway0
docker compose rm -f gateway0
docker compose up -d gateway0
```

### 步骤5: 清除Redis缓存

这是**最关键**的一步：

```bash
# 删除权限缓存
docker exec redis redis-cli -a primihub DEL "sys_auth:bfs_list"

# 删除所有用户登录token（强制用户重新登录）
docker exec redis redis-cli -a primihub KEYS "sys_user:login_token_*" | \
  xargs -I {} docker exec redis redis-cli -a primihub DEL {}
```

### 步骤6: 用户重新登录

用户必须执行以下操作：
1. **退出系统**
2. **清除浏览器缓存**（或使用Ctrl+F5强制刷新）
3. **重新登录系统**
4. **访问新模块**

## 常见问题

### Q1: 重启容器后仍然404

**A**: 确保已清除Redis缓存，并且用户已重新登录获取新token。

### Q2: 为什么要删除用户登录token？

**A**: 因为用户登录时，权限列表会被缓存在token对应的Redis key中。新增权限后，旧token中不包含新权限，必须重新登录才能获取完整的权限列表。

### Q3: 如何避免这个问题？

**A**: 在开发环境中，每次添加新模块后，按顺序执行以下操作：
1. 数据库脚本（添加权限、表结构等）
2. 后端代码（Controller、Service等）
3. 网关配置（application.yaml路由）
4. Maven编译
5. Docker镜像构建
6. 容器重启
7. Redis缓存清除
8. 用户重新登录

### Q4: 生产环境如何平滑更新？

**A**:
1. 提前在数据库中添加新权限，但设置`is_show=0`（隐藏）
2. 部署新版本镜像，使用灰度发布
3. 清除Redis权限缓存
4. 设置`is_show=1`启用新功能
5. 通知用户重新登录

## 涉及的关键文件

### 后端
- `primihub-service/gateway/src/main/resources/application.yaml` - Gateway路由配置
- `primihub-service/gateway/src/main/java/com/primihub/gateway/filter/SysAuthGatewayFilterFactory.java` - 权限过滤器
- `primihub-service/biz/src/main/java/com/primihub/biz/service/sys/SysAuthService.java` - 权限服务
- `primihub-service/application/src/main/java/com/primihub/application/controller/sys/*Controller.java` - 各模块Controller

### 数据库
- `sys_auth` - 权限表
- `sys_ra` - 角色权限关联表
- Redis: `sys_auth:bfs_list` - 权限树缓存
- Redis: `sys_user:login_token_*` - 用户登录token

### 前端
- `primihub-webconsole/src/router/index.js` - 前端路由
- `primihub-webconsole/src/api/*.js` - API调用

## 技术原理

### 权限验证流程

```
浏览器 -> Nginx -> Gateway -> Application
              |
              +-> SysAuthGatewayFilterFactory
                    |
                    +-> 读取token中的用户信息
                    +-> 从Redis/DB获取权限URL映射
                    +-> 检查当前URL是否需要权限
                    +-> 验证用户是否有该权限
                    +-> 通过则放行，否则返回NO_AUTH
```

### 权限缓存机制

1. **第一次请求**: `getSysAuthForBfs()` 从数据库读取所有权限，存入Redis的`sys_auth:bfs_list`
2. **后续请求**: 直接从Redis读取权限列表，不访问数据库
3. **权限更新**: 调用`sysAuthPrimaryRedisRepository.deleteSysAuthForBfs()`删除缓存，下次请求重新加载

### 为什么需要重启Gateway？

Gateway在启动时会加载路由配置（`application.yaml`），这些路由信息不是从Redis读取的，而是在代码中硬编码的。因此修改路由配置后必须重启Gateway才能生效。

## 快速修复脚本

创建一个快速修复脚本 `fix-new-module-404.sh`:

```bash
#!/bin/bash

echo "开始修复新模块404问题..."

# 1. 重新编译
cd /home/primihub/primihub-platform/primihub-service
echo "正在重新编译..."
mvn clean install -DskipTests -pl biz,application,gateway -am

# 2. 重新构建镜像
echo "正在重新构建Docker镜像..."
docker build -t 192.168.99.10/primihub/privacy:1.8.0 -f Dockerfile.ubuntu .

# 3. 重启容器
cd ~/github/primihub-deploy/docker-all-in-one
echo "正在重启容器..."
docker compose stop application0 gateway0
docker compose rm -f application0 gateway0
docker compose up -d application0 gateway0

# 4. 等待容器启动
echo "等待容器启动..."
sleep 20

# 5. 清除Redis缓存
echo "正在清除Redis缓存..."
docker exec redis redis-cli -a primihub DEL "sys_auth:bfs_list"
docker exec redis redis-cli -a primihub KEYS "sys_user:login_token_*" | \
  xargs -I {} docker exec redis redis-cli -a primihub DEL {}

echo "修复完成！请用户重新登录系统。"
```

使用方法：
```bash
chmod +x fix-new-module-404.sh
./fix-new-module-404.sh
```

## 总结

新增模块后出现404的根本原因是**多层缓存不一致**：
- Gateway容器使用旧路由配置
- Application容器使用旧JAR文件
- Redis缓存了旧的权限数据
- 用户session中有旧的权限列表

解决方案的核心是：**重新编译 -> 重建镜像 -> 重启容器 -> 清除缓存 -> 用户重新登录**

记住这个口诀：**增模块，改路由，必重启；清缓存，重登录，才生效。**
