# 白名单404问题完整分析报告

## 问题现象

用户点击"白名单列表"按钮时，浏览器返回：**请求出错(404)**

即使已经：
- ✅ 后端Controller代码正确（WhitelistController.java）
- ✅ 数据库表已创建（whitelist表）
- ✅ Gateway路由配置正确（whitelist-url-proxy）
- ✅ 权限已分配给admin角色（sys_ra表）
- ✅ Docker镜像已重建
- ✅ 容器已重启

但问题仍然存在！

## 根本原因

这是一个**多层级配置错误**的问题，涉及两个关键配置：

### 问题1：权限URL配置错误（已修复）

**数据库表：`sys_auth`**

| 字段 | 说明 | 错误配置 | 正确配置 |
|------|------|----------|----------|
| `full_path` | 前端路由路径 | `/whitelist/list` | `/whitelist/list` |
| `auth_url` | **后端API路径** | ❌ `/whitelist/list` | ✅ `/whitelist/findWhitelistPage` |

**为什么`auth_url`必须是后端API路径？**

查看Gateway权限验证代码（`SysAuthGatewayFilterFactory.java:60-66`）：

```java
Map<String, SysAuthNodeVO> mapping = sysAuthService.getSysAuthUrlMapping();
SysAuthNodeVO sysAuthNodeVO = mapping.get(rawPath); // rawPath是后端API路径！
if (sysAuthNodeVO != null) {
    if (!sysUserListVO.getAuthIdList().contains(sysAuthNodeVO.getAuthId().toString())
        &&!authIdList.contains(sysAuthNodeVO.getAuthId())) {
        return GatewayFilterFactoryTool.writeFailureJsonToResponse(exchange, BaseResultEnum.NO_AUTH);
    }
}
```

**权限验证流程**：

1. 用户请求：`GET /prod-api/whitelist/findWhitelistPage`
2. Nginx转发到Gateway：`GET /whitelist/findWhitelistPage`
3. Gateway获取URL路径：`rawPath = "/whitelist/findWhitelistPage"`
4. 在权限映射中查找：`mapping.get("/whitelist/findWhitelistPage")`
5. 如果`auth_url`配置的是`/whitelist/list`，则**找不到匹配**，返回404

### 问题2：Redis权限缓存未刷新（已修复）

即使修复了数据库配置，Gateway使用的是**Redis缓存**中的权限数据：

**Redis Key: `sys_auth:bfs_list`**

这个缓存由`SysAuthService.getSysAuthForBfs()`方法维护（第129-135行）：

```java
public List<SysAuthNodeVO> getSysAuthForBfs() {
    List<SysAuthNodeVO> list = sysAuthPrimaryRedisRepository.getSysAuthForBFS();
    if(list==null||list.size()==0) {
        list = sysAuthSecondarydbRepository.selectAllSysAuthForBFS();
        sysAuthPrimaryRedisRepository.setSysAuthForBFS(list);
    }
    return list;
}
```

**缓存刷新时机**：
- 创建/修改/删除权限时会调用`deleteSysAuthForBfs()`清除缓存
- 但**直接UPDATE数据库不会清除缓存**！

## 完整的权限验证流程

### 1. 用户登录流程（SysUserService.java:124-150）

```java
public BaseResultEntity baseLogin(SysUser sysUser,String ip){
    // 1. 获取用户的角色ID列表
    Set<Long> roleIdSet = Stream.of(sysUser.getRoleIdList().split(","))
        .filter(item->!"".equals(item))
        .map(item->(Long.parseLong(item)))
        .collect(Collectors.toSet());

    // 2. 从数据库查询角色对应的所有权限ID
    Set<Long> authIdList = sysRoleSecondarydbRepository.selectRaByBatchRoleId(roleIdSet);

    // 3. 从Redis/DB获取所有权限节点
    List<SysAuthNodeVO> roleAuthRootList = sysAuthService.getSysAuthForBfs();

    // 4. 筛选出用户拥有的权限
    List<SysAuthNodeVO> grantAuthRootList = roleAuthRootList.stream()
        .filter(item->authIdList.contains(item.getAuthId()))
        .map(item->{item.setIsGrant(1); return item;})
        .collect(Collectors.toList());

    // 5. 将权限ID列表转为字符串，存入token
    String authIdListStr = authIdList.stream()
        .map(String::valueOf)
        .collect(Collectors.joining(","));
    sysUserListVO.setAuthIdList(authIdListStr);

    // 6. 生成token并保存到Redis
    String token = PlatformHelper.generateOwnToken(SysConstant.SYS_USER_TOKEN_PREFIX, sysUser.getUserId(), date);
    sysUserPrimaryRedisRepository.updateUserLoginStatus(token, sysUserListVO);

    return BaseResultEntity.success(map);
}
```

### 2. Gateway权限验证流程（SysAuthGatewayFilterFactory.java:38-76）

```java
public GatewayFilter apply(Object config) {
    return (((exchange, chain) -> {
        // 1. 获取token
        String token = exchange.getAttribute(BaseParamEnum.TOKEN.getColumnName());

        // 2. 从Redis获取用户信息（包含authIdList）
        SysUserListVO sysUserListVO = sysUserPrimaryRedisRepository.findUserLoginStatus(token);

        // 3. 从Redis/DB获取权限URL映射
        Map<String, SysAuthNodeVO> mapping = sysAuthService.getSysAuthUrlMapping();

        // 4. 获取当前请求的URL路径
        String rawPath = exchange.getRequest().getURI().getRawPath();

        // 5. 在权限映射中查找当前URL对应的权限节点
        SysAuthNodeVO sysAuthNodeVO = mapping.get(rawPath);

        // 6. 如果找到权限节点，验证用户是否拥有该权限
        if (sysAuthNodeVO != null) {
            if (!sysUserListVO.getAuthIdList().contains(sysAuthNodeVO.getAuthId().toString())
                && !authIdList.contains(sysAuthNodeVO.getAuthId())) {
                // 用户没有该权限，返回403
                return GatewayFilterFactoryTool.writeFailureJsonToResponse(exchange, BaseResultEnum.NO_AUTH);
            }
        }

        // 7. 权限验证通过，转发请求到application
        return chain.filter(exchange.mutate().request(newRequest).build());
    }));
}
```

## 修复步骤总结

### 步骤1：修复数据库权限URL配置

```sql

UPDATE sys_auth SET auth_url = '' WHERE auth_code = 'Whitelist';
UPDATE sys_auth SET auth_url = '/whitelist/findWhitelistPage' WHERE auth_code = 'WhitelistList';

```

### 步骤2：清除Redis权限缓存

```bash
docker exec redis redis-cli -a primihub DEL "sys_auth:bfs_list"
```

### 步骤3：重启Gateway（清除内存缓存）

```bash
cd ~/github/primihub-deploy/docker-all-in-one
docker compose restart gateway0
```

### 步骤4：清除所有用户登录token（强制重新登录）

```bash
docker exec redis redis-cli -a primihub KEYS "sys_user:login_token_*" | \
  xargs -I {} docker exec redis redis-cli -a primihub DEL {}
```

### 步骤5：用户重新登录

用户重新登录后：
1. 系统从数据库查询最新的权限列表（包含Whitelist权限）
2. 生成包含新权限的token
3. 首次请求时，Gateway从数据库重新加载权限映射到Redis
4. 权限验证通过，请求成功

## 关键要点

### 1. auth_url vs full_path

| 字段 | 用途 | 示例 | 使用场景 |
|------|------|------|----------|
| `full_path` | 前端路由 | `/whitelist/list` | Vue Router路由配置 |
| `auth_url` | 后端API路径 | `/whitelist/findWhitelistPage` | Gateway权限验证 |

### 2. 多层缓存

1. **数据库**：`sys_auth`表 → 最终数据源
2. **Redis缓存**：`sys_auth:bfs_list` → Gateway读取
3. **Gateway内存**：可能有Java对象缓存
4. **用户Token**：`sys_user:login_token_*` → 包含authIdList

**所有缓存必须同步更新，否则会出现404！**

### 3. 权限验证的双重检查

Gateway会检查两个条件（满足其一即可）：

```java
if (!sysUserListVO.getAuthIdList().contains(sysAuthNodeVO.getAuthId().toString())  // 用户直接权限
    && !authIdList.contains(sysAuthNodeVO.getAuthId())) {  // 角色权限
    return 403; // 无权限
}
```

1. **用户直接权限**：token中的authIdList（登录时从sys_ra查询）
2. **角色权限**：实时从sys_ra查询（根据用户角色）

## 常见错误

### ❌ 错误1：auth_url配置为前端路由

```sql
-- 错误
UPDATE sys_auth SET auth_url = '/whitelist/list' WHERE auth_code = 'WhitelistList';
```

结果：Gateway找不到`/whitelist/findWhitelistPage`的权限映射，返回404

### ❌ 错误2：只修改数据库，不清除Redis

```sql
UPDATE sys_auth SET auth_url = '/whitelist/findWhitelistPage' WHERE auth_code = 'WhitelistList';
```

结果：Gateway仍然从Redis读取旧配置，返回404

### ❌ 错误3：只清除Redis，不重启Gateway

```bash
docker exec redis redis-cli -a primihub DEL "sys_auth:bfs_list"
```

结果：Gateway可能有内存缓存，返回404

### ❌ 错误4：只重启Gateway，不清除用户token

```bash
docker compose restart gateway0
```

结果：旧token中的authIdList可能不包含新权限，返回403

## 结论

白名单404问题的根本原因是：**auth_url配置错误 + 多层缓存未刷新**

修复的核心是：
1. 确保`auth_url`配置为**后端API路径**
2. 清除**所有层级的缓存**（数据库 → Redis → Gateway内存 → 用户Token）
3. 用户**重新登录**获取新权限

记住这个口诀：**auth_url配后端，缓存要清完，用户需重登！**
