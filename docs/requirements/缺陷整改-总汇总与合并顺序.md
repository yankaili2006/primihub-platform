## 二、推荐合并顺序

**8 个功能分支互相独立**（仅 T6/T10 共改一个文件，见下），顺序主要为降低评审/冲突成本、并让"底座先落地"：

1. **T4+T5（SQL）** — 数据层地基（权限/表），其它修复隐含依赖表与权限已就位
2. **T3（网关）** — 基础设施路由，独立
3. **T1（returnCode）** — 前端隔离改动，无冲突
4. **T7+T8+T9** — 前后端混合，独立
5. **T6 → T10**（⚠️ 有先后）— **两者都改 `ApiManageService.java`**：
   - T6 改 `setApiMethod`（第 63、95 行，容错读 `method`）
   - T10 重写 `exportApiLog`（+ imports + `writeExportError`）
   - 不同 hunk，大概率自动合并；**建议先合 T6 再合 T10**，若冲突只需保留两处各自改动
6. **T2** — 前端接线，最后合（依赖 T3 让模块可达、T4 让菜单/权限出现）
7. **docs** — 任意时间合

> **文件重叠仅一处**：`ApiManageService.java`（T6 与 T10）。其余零重叠，可并行评审。

## 五、覆盖与遗留

**已覆盖 ~40/44 缺陷。** 剩余均为"后端功能缺失"，属新开发（非改 bug），建议单独立项：

| 遗留 | 原因 | 建议 |
|---|---|---|
| 警务融合 / 电子证照 **日志记录 + 日志导出** | 两模块 log 端点后端完全缺失 | 补 SceneLog 控制器 + 表 + 导出 |
| 警务融合 **batchExchange** | 无 `/policeFusion/exchange` | 补批量交换端点 |
| 联邦学习 **dataMerge / modelImport** | 无合并 / 上传（multipart）后端 | 补单方合并、模型导入端点 |
| 联邦学习 **index 启动任务** | 无 start 端点（createTask 时自动运行） | 产品确认是否需显式启动 |
| **时间戳申请 evidenceId 语义** | 前端 FormData(file/data) vs 后端只认 evidenceId，语义错位 | 产品定义"申请=新建存证并打时间戳"或"对已有存证补时间戳"后再实现 |
| 各"导出历史"列表 | 无历史查询端点 | 补历史表 + 分页查询（当前客户端构建，动作已真实化） |

## 六、方法论说明（诚实边界）

- **SQL（T4/T5）**：用真实 MySQL8 容器验证加载 + 幂等，可信度最高。
- **后端 Java（T6/T7/T8/T10）**：`mvn -o compile` 主代码 BUILD SUCCESS，逻辑对照 schema/mapper 核对；受限于无整套运行环境，未做端到端联机验证。
  - 注：`biz` 存在**与本次无关的既有坏测试** `FederatedQueryServiceImplTest:160`（'void' type not allowed），阻塞 `test-compile` 但不影响主代码编译与部署，未纳入本次。
- **前端（T1/T2）**：改动文件 eslint 全过；导出/下载走同仓已验证的 blob 模式；受限于无法在此运行前端，未做浏览器端联机验证。个别 mock 页后端字段与模板不完全一致处做了**防御性映射**（真实数据优先、缺失字段降级为空而非崩溃）。
- **原则**：后端缺失的页**一律不硬接**，避免制造"假成功"。
